# 架构认知

> 从项目中蒸馏出的通用架构原则。不绑定任何具体项目。

## 核心关注

- **系统设计模式**：热重载、模板系统、双轨加载、保活与业务解耦、集群状态归属、配置热同步
- **数据流设计**：类型契约归一化、队列 envelope/payload 分离、派生产物单源、叶子字段解码边界
- **模块边界**：依赖方向、接口设计、解耦策略、不可逆操作延迟
- **并发与异步模式**：背压机制、确认语义、控制流拆分、membership 保活
- **鉴权与安全**：JWT 生命周期、密码哈希、防爆破、令牌失效面
- **分布式系统**：乐观锁、配置热同步、集群限频、分库分表

## system-design/ — 系统设计原则

- **背压必须在管道入口** — auto-commit 切断反馈回路；信号量挡在 consume loop 是最可靠的背压原语
- **保活循环不能和慢业务绑死** — Kafka consumer poll、分布式锁续租、heartbeat 必须独立于业务执行
- **并发消费 ≠ 精确 at-least-once** — commit 提交最大 offset；at-most-once + 幂等已等价
- **双循环是过度工程信号** — 生命周期不独立的循环应合并；信号量 + TaskGroup 足够
- **集群限频状态必须在共享存储** — 只共享配置阈值不共享运行状态，不构成全局约束
- **不可逆操作延迟到确定性最高点** — Pipeline 入口不做 DELETE；原子替换在确定性最高的替换事务里
- **配置热同步：拉取 > 通知** — 定期拉取 + 版本比对 + 优雅降级，比 pub/sub 更可靠
- **超时策略按操作语义拆分** — connect/read/write/upload/download/parse 各独立预算
- **运行时不变式放统一出口** — 分支产物契约一致，出口校验+去重，不掩盖上游证据缺失
- **容器日志 vs 应用日志是两条边界** — stdout/容器驱动/本地文件三层各管各的生命周期
- **JWT 是声明系统，不是完整会话** — 签名验证 ≠ 传输安全 ≠ 防盗用 ≠ 撤销 ≠ 续期
- **分库分表的本质矛盾**：物理切分 vs 多维查询 — 分片键决定能力边界，切分策略不是核心，查询聚合才是
- **扩容不如控量** — 冷热分离 + 归档让热库数据量可控；终局是分布式数据库

## data-flow/ — 数据流与契约设计

- **叶子字段契约在解码边界归一化** — LLM 输出不稳定，入口 prompt 降错误率，解码边界修形状，最终模型拦漏网
- **抽取层级对象：标题角色 > 标题层级** — 集合名负责切 range，实例名负责落字段
- **队列消息区分 envelope 和 payload** — 调度元数据在 Worker 边界剥离，不污染业务函数签名
- **滑动窗口按时间定义成员资格** — 惰性剪枝比维护窗口大小更优雅；deque + 时间剪枝 = O(1) 两端操作
- **派生产物共享同一个中间事实** — 原始输入只解析一次；后续再解析不是"复用源数据"，是制造并行宇宙
