# 编码技巧

> 语言层面的洞察、惯用法、反模式。可跨语言迁移的思维方式。

## 核心关注

- Python 惯用法（类型系统、数据结构选择、Pydantic 分层、模板引擎）
- 通用编码模式（哨兵值语义、生命周期归属、单一事实源、覆盖率层级）
- 反模式识别（类型契约分散、结构占位污染、算法复用与调度归属混淆）

## 已沉淀

### Python 模式
- **Pydantic `mode="before"` 是数据清洗层** — 归一化输入形状；`mode="after"` 做业务校验；不要混用
- **Jinja2 模板 > f-string 拼接** — 任何结构化文档生成场景；需显式配置 `select_autoescape()`
- **Redis ZSet 是轻量级单维时序索引** — 适合 < 1万点/天 + 简单范围查询；不适合多标签/高基数/长期保留
- **滑动窗口：Deque + 时间剪枝** — 惰性剪枝比固定容量更优雅；`maxlen` 做异常流量兜底

### 通用编码模式
- **字段类型契约必须只有一个事实源** — 注册表定义契约，Prompt 引导遵守，assembler 只执行；不能散落在自然语言和 if 里
- **派生产物共享同一个中间事实** — 原始输入只解析一次；携带 identity 的中间对象是唯一真相源
- **结构占位服从字段业务语义** — "对齐载体"和"业务覆盖层"不能混用同一种哨兵值（`{}` vs `null`）
- **生命周期对象不能临时 new** — 状态对象由 owner 创建、consumer 消费；consumer 自己创建 = 把重置 bug 藏进局部变量
- **算法复用和 task owner 是两个维度** — 能力不再通用时，拆成 support module + thin owner wrappers，不要复制算法
- **覆盖率要定义层级** — 代码触达 → 对象赋值 → 非空收集 → 持久化事件是四个不同层；测试锁定最终业务消费层
- **正则召回 ≠ 语义归属** — 规则做可解释召回，模型做语义归属审核，写入层保持稳定 schema
- **相似度阈值先确认分数方向** — 越小越近 vs 越大越相关；用真实日志 + 搜索参数确认方向
- **多 Worker 聚合必须加权** — 响应时间按任务数加权，成功率按请求数加权，不能对指标值直接平均
